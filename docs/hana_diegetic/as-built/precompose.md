# Precompose (LDR)

## What it is

`precompose_ldr()` flattens a layout-element subtree through an offscreen LDR render target and draws the cached result as a single alpha-blended image in the source panel. It originates from the HDR-text discussion: some subtrees should composite as one flat SDR image rather than having each descendant (text SDF, fills, borders, shapes) draw individually through the normal analytic path under an HDR scene camera. Calling `El::precompose_ldr()` (or `Text::precompose_ldr()`) marks an element boundary; the subtree beneath it is rendered once into an owned `Image` by a hidden helper panel + orthographic camera, and the source panel emits a single `RenderCommandKind::PrecomposeLdr` command that draws that image where the boundary element sits.

## How it works

**Identity (`PanelElementId`, `crates/hana_diegetic/src/ime/ids.rs`).** One panel-local identity type, `enum PanelElementId { Named(String), Auto(AutoElementId) }`, re-exported from the crate root. `El::id(...)`, `Text::id(...)`, and `El::editable_field(field_id, ...)` (`layout/builder.rs`) all take `impl Into<PanelElementId>` and share the single namespace. Only `Named` is publicly constructible (`named`, `From<&str>`, `From<String>`); `Auto` is minted crate-internally by `PanelElementId::auto(u32)` through the builder's `take_auto_id` (`layout/builder.rs:874`), which bumps a per-build order counter for every unnamed text run. Because no public constructor yields `Auto`, named and auto ids cannot collide by construction.

**Builder flag.** `precompose_ldr()` sets `common.precompose = PrecomposeMode::Ldr` on `El` (`layout/builder.rs:619`) or on the text leaf's layout (`:165`). Default is `PrecomposeMode::Direct`. `PrecomposeMode` is `pub(crate)` in `layout/element.rs`.

**Engine emission (`layout/engine/positioning.rs`).** During both the positioning traversal and `render_commands_from_geometry`, when `element.precompose == PrecomposeMode::Ldr` the engine calls `emit_precompose_command` (`:303`) to push one `RenderCommandKind::PrecomposeLdr` at the element's bounds and z-index, and then does NOT descend into or emit any commands for that element's subtree. The boundary is a hard stop in the command stream.

**Subtree cloning (`layout/element.rs`).** `LayoutTree::precompose_subtree(root_index, width, height)` (`:333`) clones the boundary element and its descendants into a standalone tree via `clone_subtree_into`, which sets each clone's `precompose = PrecomposeMode::Direct` (so the helper never recursively precomposes), clears per-text HDR coverage bias (`config.clear_hdr_text_coverage_bias()`), and on the root fixes `width`/`height` to `Sizing::Fixed` from the passed dimensions.

**Render command classification (`layout/render.rs`).** `RenderCommandKind::PrecomposeLdr` (`:110`) maps to `DrawSortTier::Surface` (`:131`) â€” same tier as rectangles/borders/images â€” and `draw_batch_family()` (`:145`) routes it to `DrawBatchFamily::Image`, so it draws through the batched image family exactly like `Image` (see `as-built/image-batching.md`).

**Render side (`render/precompose.rs`).** `ensure_panel_precompose_caches` (`:52`, runs on `Changed<ComputedDiegeticPanel>`) drives everything:
- `collect_precompose_commands` (`:132`) gathers `(element_idx, bounds)` for each `PrecomposeLdr` command.
- `remove_stale_entries` (`:461`) despawns helper panel + camera and retires the image for element indices no longer present.
- For each live boundary: `precompose_subtree` (`:142`) builds the cloned tree; `precompose_pixel_size` (`:160`) â†’ `precompose_axis_pixels` (`:167`) computes the target `UVec2`; `helper_panel_for_subtree` (`:178`) builds a hidden `DiegeticPanel::world()` sized in points, `Anchor::Center`, `Lighting::Unlit`, `HdrTextCoverageBias::NO_BIAS`, forwarding overridden materials/shadow-casting/alpha-mode from the source panel.
- `sync_entry` (`:219`) either updates an existing `PrecomposeCacheEntry` (`update_existing_cache_entry`, `:269`) or spawns a new helper panel + camera as children of the source panel (`spawn_precompose_entry`, `:358`). The camera (`precompose_camera`, `:424`) is a `Camera3d` with `ClearColorConfig::Custom(Color::NONE)`, order `-100`, viewport = pixel size, targeting the owned image; the projection (`precompose_projection`, `:447`) is orthographic `ScalingMode::FixedVertical { viewport_height: bounds.height }`. Helpers are placed on `RenderLayers::layer(30)` and spatially separated by `helper_origin` (`:457`, `PRECOMPOSE_HELPER_SPACING = 10000.0 * element_idx`).
- The owned `Image` (`precompose_image`, `:413`) is `TextureFormat::Bgra8UnormSrgb` (LDR) with linear sampling.
- `PanelPrecomposeCache` (`panel/precompose.rs`) holds `entries: HashMap<usize, PrecomposeCacheEntry>` keyed by source `element_idx`, plus retirement/activation queues. `cleanup_retired_precompose_images` (`:106`) and `activate_pending_precompose_cameras` (`:119`) defer image removal and camera reactivation by a frame so Bevy's render-world target bookkeeping settles.

**Draw in source panel (`render/image_batch.rs`).** `route_image_batch_records` (`:498`, ordered `.after(cleanup_retired_precompose_images)`) handles `PrecomposeLdr` in `image_record_source` (`:596`) by looking up `precompose_cache.entry(command.element_idx)` â€” skipping the command if the entry is absent, never synthesizing a default handle â€” and emitting a `ResolvedImageRecord` with the cached `entry.image`, white tint, and the boundary bounds into the batched image family. It draws exactly like a normal panel image.

## Invariants

- **Named ids are unique panel-wide.** `LayoutTree::duplicate_named_element_id` (`layout/element.rs:623`) scans element ids and editable-field ids together and returns the first `Named` id seen twice; `DiegeticPanelBuilder::build` rejects it. Auto ids are skipped (unforgeable/unique by construction). Reusing a named id anywhere in the panel tree is a build error.
- **The boundary stops the normal stream.** For a `PrecomposeLdr` element the engine emits exactly one command and no descendant text/rectangle/border/shape commands. Locked by `precompose_ldr_emits_single_boundary_command_without_descendants` (`layout/engine/integration_tests.rs:225`): count of `PrecomposeLdr` == 1, and no `Text`/`Rectangle` commands exist.
- **Clone clears modes and fixes root size.** `precompose_subtree` must keep setting descendant `precompose` to `Direct` (no recursive helpers) and fixing the cloned root to the boundary size. Locked by `precompose_subtree_clears_modes_and_fixes_root_size` and `precompose_ldr_sets_element_mode` (`layout/element.rs:1137`, `:1158`).
- **Flat image, not a scene graph.** The output is a single alpha-blended image. It does NOT preserve per-descendant depth, ordering against scene geometry, or interactive descendants â€” no hit-testing/IME inside a precomposed subtree.
- **One image owned per source element, reused/resized by pixel size.** The cache key is the source `element_idx`; the allocation is reused while `entry.pixel_size == pixel_size` and replaced (old handle retired) when it differs (`update_existing_cache_entry:276`). Locked by `cache_sync_preserves_image_when_precompose_bounds_are_unchanged` and `cache_sync_spawns_helper_panel_camera_and_image` (`render/precompose.rs:610`, `:528`).
- **Helper tree tracks content at fixed bounds.** A same-bounds text change refreshes the helper panel's tree without reallocating the image: `cache_sync_refreshes_helper_tree_when_text_changes_with_same_bounds` (`:669`). Replacing a helper's tree invalidates the scaled-tree cache: `precompose_helper_replace_invalidates_scaled_tree_cache` (`panel/diegetic_panel.rs:1748`) â€” after `replace_from_precompose_helper`, `get_or_update` misses (does not serve a stale scaled tree).
- **No public precompose enum.** Only `precompose_ldr()` is public. `PrecomposeMode` stays crate-internal; do not expose it until a second mode with real semantics exists.

## Calibration / gotchas

- **Pointsâ†’pixels rounding.** `precompose_axis_pixels` (`render/precompose.rs:167`) computes `ceil(points * PRECOMPOSE_SUPERSAMPLE)` with `PRECOMPOSE_SUPERSAMPLE = 8`, clamped to `[MIN_PRECOMPOSE_PIXELS(=1), u32::MAX]`. Non-finite or non-positive extents fall back to 1 pixel. So the target is 8Ă— supersampled per axis; changing the supersample factor changes VRAM cost quadratically.
- **LDR target.** The image is `Bgra8UnormSrgb` (8-bit sRGB, not HDR/float). Composited output is clamped to SDR range â€” that is the point (SDR text edges under an HDR scene) but means the precomposed result cannot carry HDR values back into the parent, and it draws with `Color::WHITE` tint (no HDR boost applied at draw time).
- **Projection/bounds edge cases.** `precompose_projection` and `helper_panel_for_subtree` guard height with `.max(MIN_PRECOMPOSE_PIXELS)`, and `precompose_image`/camera viewport clamp to `â‰Ą 1` (`.max(UVec2::ONE)`). Zero/degenerate bounds still produce a valid 1px target rather than panicking. `precompose_camera` currently ignores its `_bounds` argument â€” sizing comes entirely from `pixel_size` (viewport) and the projection's `FixedVertical`.
- **Reuse-vs-resize boundary.** The decision is `entry.pixel_size != pixel_size` (post-rounding). Because pixel size is `ceil`'d, sub-pixel bounds jitter that rounds to the same integer size reuses the image (no realloc, no camera re-activation). Crossing an integer pixel boundary retires the old image, allocates a new one, and defers camera reactivation one frame (`defer_camera_activation`) so the new target propagates before the camera renders â€” expect a one-frame settle on resize.
- **Frame-deferred cleanup.** Retired images and pending camera activations survive one extra frame (`drain_ready_*` gate on `frames > 1`). Do not assume an old render target is freed the same frame its boundary changes.

## Why

- **LDR-first, no public enum.** The concrete need was SDR text edges under an HDR camera; there is no second precompose mode with real semantics yet. Exposing an enum now would freeze an API around a single member. `precompose_ldr()` is the whole public surface until a real second mode arrives; `PrecomposeMode { Direct, Ldr }` stays internal.
- **One shared `PanelElementId` namespace.** Element ids, text ids, and editable-field ids are all panel-local semantic identity. Collapsing them into one type/namespace lets a container and a child text carry distinct ids, lets precompose/text-lookup/hit-testing/IME routing all key off the same identity, and makes duplicate detection a single scan. Auto ids being unforgeable keeps builder-minted positional ids from ever colliding with author names, so the shared namespace is safe without prefixing.
- **Flat image tradeoff.** Rendering the subtree offscreen and compositing one quad is what makes the SDR flattening possible, but it discards per-descendant depth, scene-geometry interleaving, and interactivity. That loss is accepted deliberately: precompose is for subtrees that should read as one flat surface (badges, glyph clusters, decorative panels), not for interactive or depth-sorted content. Callers who need those properties must not precompose.
