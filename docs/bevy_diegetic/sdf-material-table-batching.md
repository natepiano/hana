- **Invariants:**
  - **One implementation target.** There is no SDF-local material interner and no shipped record-local SDF color implementation. SDF/fill batching uses the frame material table from its first production route.
  - **The frame-table SDF path is the shipped route.** The legacy `PanelSdfMesh` / `PanelSdfSurface` / `spawn_sdf_quad` route is gone; the remaining `PanelInteractionMesh` is picking-only.
  - **SDF is for panel fills and borders only.** There is no callout SDF render route and no analytic callout adapter; callout capability returns later on the PanelShape path (Phase 9+). Do not add new SDF primitive forms for callouts, arrows, caps, circles, triangles, or diamonds.
  - **Simple material table first.** The production plan starts with a frame-built dense material table, not a persistent slot allocator. Each frame builds the current material rows and the matching draw records together, then measures the cost. The durable allocator/retirement model is documented only as an appendix alternative and is not part of the active implementation plan.
  - **Old interner cleanup is complete.** `VisualMaterialInterner`, scalar-value `BaseMaterialId` batching, and `VisualBatchKey` must stay absent from production and probe-side analytic material routing.
  - **Relationships are source traversal, not slot ownership.** Use Bevy relationships only where the render source already is an entity, matching the existing `TextRunOf` / `PanelTextRuns` pattern. The frame material table is plain render data for the current frame; do not create material-slot entities, per-SDF-surface entities, or per-panel-shape-primitive entities just to use relationships.
- **Material-as-data.** Batch keys carry no scalar/vector PBR values. `MaterialSlotValues` is the generic per-record PBR value payload: base color including its alpha channel, emissive, metallic, perceptual roughness, reflectance, transmission, thickness, attenuation, clearcoat, anisotropy, `ior`, the `uv_transform` UV affine, and any other supported scalar/vector `StandardMaterial` fields live in the material-table row selected by `MaterialSlotId`. `AlphaMode` is not the same thing as base-color alpha; it is compatibility data for SDF, text, and panel-shape render families. Two fills/text runs/panel-shape runs differing only in table values share a batch.
- **Diegetic draw order owns depth.** Source-material `StandardMaterial::depth_bias` is not a material-table value and is not an authored material splitter in this plan. Current panel SDF construction overwrites the base material's `depth_bias` from `DrawCommandDepth`; the batched path keeps depth/order data in `DrawOrder`, `SdfRenderRecord`, `PathRenderRecord`, and batch sort fields.
- **Draw-order definitions.** `DrawZIndex` is the authored per-command z bucket. `DrawZIndexRank` is the dense rank of each distinct `DrawZIndex` in one panel's sorted z-index set; all commands with the same authored z-index share the rank. `DrawOrderIndex` is the dense per-command rank after sorting `(DrawZIndex, DrawSortTier, CommandIndex)`. `ScreenDepthBias` is derived from `DrawZIndexRank`, so Bevy's integer hardware depth-bias path advances once per z-index band, not once per command. `ClipDepthNudge` and `OitDepthOffset` are derived from `DrawOrderIndex`, so shader and OIT ordering still separate commands inside one z-index band.
- **Individual `StandardMaterial` sources are first-class.** Text runs, SDF roles, and panel-shape primitives may each resolve from distinct `StandardMaterial` sources. They still share a batch when those sources differ only by scalar/vector table values such as base color, emissive, metallic, perceptual roughness, reflectance, or `ior`. They split when the resolved material changes pipeline/resource compatibility that the renderer honors, such as alpha mode, texture resources, sampler/bind-group requirements, shader-def-driving flags, culling/sidedness, lighting mode, or shadow/prepass participation.
- **SDF exposes render policy instead of forcing it.** The SDF material ladder may provide SDF-friendly defaults (`AlphaMode::Blend`, `double_sided = true`, `cull_mode = None`), but authored `AlphaMode`, `double_sided`, and `cull_mode` remain user-controlled artistic choices. SDF batches split on those fields when they differ, just like text and panel-shape batches.
- **Material source ladders are explicit.** Each draw family owns a named material-source ladder; do not infer one family from another. Each ladder is cascade-shaped: local override, else panel override, else global/default.
  - Text resolves `TextStyle::with_material(Handle<StandardMaterial>)` (local override), else the panel-level `DiegeticPanelBuilder::text_material`, else `default_panel_material()`, plus `TextStyle` scalar/policy overrides on the resolved material. The text renderer does not read `El::material` as a text material source.
  - SDF backgrounds/borders resolve `El::material`, else `DiegeticPanelBuilder::material`, else `default_panel_material()`.
  - Panel shapes resolve a shape-local material override (`PanelLine::material` / `LineStyle::material` / `PanelCircle::material`), else the panel-level `DiegeticPanelBuilder::shape_material`, else `default_panel_material()`, plus per-shape color authoring as a base-color override on the resolved material.
  `El::material` is the source for element surfaces; using it for text or shape material must be a deliberate documented API choice, not renderer-side inference.
- **Material ladders use the cascade.** Generalize the existing cascade from `Copy` attributes to `Clone` attributes and use `Handle<StandardMaterial>` as the cascaded material value. This keeps the rule the same as the current cascade: local override, else parent/panel override, else default. Do not cascade owned `StandardMaterial` values. Material projection into `MaterialSlotValues` reads the current `Assets<StandardMaterial>` value behind the resolved handle, so edits to a shared material asset update table rows without changing the cascade value.
- **Cascade material types are source handles only.** The names below are the only material names that should participate in the cascade:
  - `SdfMaterial(pub Handle<StandardMaterial>)` cascades the source material for SDF backgrounds, borders, and other element surfaces.
  - `TextMaterial(pub Handle<StandardMaterial>)` cascades the source material for text runs.
  - `ShapeMaterial(pub Handle<StandardMaterial>)` cascades the source material for panel-shape primitives.
  These source-handle types resolve before material-table projection. They are not Bevy render material asset types and they do not own shader bindings.
- **Render material types do not cascade.** `SdfExtendedMaterial = ExtendedMaterial<StandardMaterial, SdfExtension>` and `PathExtendedMaterial = ExtendedMaterial<StandardMaterial, PathExtension>` are GPU render material asset types. They own shader extension bindings and batch plumbing, not authored source-material identity. They may read material-table rows after source handles have been resolved and projected, but they are never cascade attributes.
- **Render material extensions are batch plumbing, not source material identity.** Text and panel shapes share `PathExtendedMaterial = ExtendedMaterial<StandardMaterial, PathExtension>`. `PathExtension` owns shader plumbing such as path/quad/run storage buffers and batch uniforms; it is not a text- or shape-source material. Do not add separate `TextExtension` and `ShapeExtension` unless a proven pipeline requirement appears. The SDF batch path uses `SdfExtendedMaterial = ExtendedMaterial<StandardMaterial, SdfExtension>`. `SdfExtension` owns the `SdfRenderRecord` buffer, frame material table buffer, and batch-level shader plumbing for panel fill/border SDFs only. Per-surface geometry/material identity lives in `SdfRenderRecord` and the frame material table. The old per-surface SDF render material was removed with the legacy SDF route.
- **Texture-backed material sampling.** Every sampled `StandardMaterial` texture channel resolved from a record's source material — `base_color_texture`, `emissive_texture`, `metallic_roughness_texture`, `normal_map_texture`, `occlusion_texture`, and any other supported channel — is sampled by all three render families (SDF fills, text glyphs, panel shapes), not only base color and not only SDF fills. Each record samples through the `StandardMaterial` half of its `ExtendedMaterial` exactly as Bevy's `pbr_input_from_standard_material` does, so the channels feed PBR the same way a normal mesh would: the SDF/glyph/stroke coverage composites the sampled base color before writing alpha, while emissive/metallic-roughness/normal/occlusion contributions flow into `apply_pbr_lighting` unchanged. All channels share one UV: the record's element-local box UV, `0..1` across the resolved layout box of the SDF surface, text run, or panel-shape silhouette, so the material maps once across the whole record and the coverage/outline/stroke stencils it. This is the same mapping SDF fills already get from the quad's `0..1` UV. Because the target text and shape render families share `PathExtendedMaterial` / `PathExtension` (see the render-extension principle above), the glyph and shape sampling is added once in the shared analytic path shader; SDF samples in `SdfExtension`, which already runs `pbr_input_from_standard_material` and therefore already applies the non-base-color channels today. The box UV is per-record render data carried in `SdfRenderRecord`, the per-glyph instance record, and `PathRenderRecord`; it is never a material-table value and never a batch-key field. Texture presence per channel, and any channel-specific mesh-attribute requirement such as tangents for `normal_map_texture`, are `ResourceCompatibility` / pipeline splitters per the Phase 2 classification, so a texture-backed record forms its own batch while the material table holds only scalar/vector values; channels that need tangents add that mesh-attribute requirement to the batch geometry. To texture a single glyph or a sub-span, author it as its own run/record. `ElementContent::Image` leaves remain a separate, unbatched compositing path; this principle is per-record material texturing of SDF/text/shape surfaces, not image-leaf batching. The record's authored `uv_transform` — a `StandardMaterial` affine (scale/offset/rotation) applied to UVs, not a texture — is supported as a per-record material-table value: each shader reads it from the record's table row and composes it with the box UV as `final_uv = uv_transform * box_uv` before sampling every channel, so two records sharing the same textures but tiling/offsetting/rotating differently still batch together and differ only in table data. An identity `uv_transform` maps each texture once across the box; tiling past `0..1` repeats only when the bound texture's sampler uses a `Repeat` address mode, which is a property of the texture resource and therefore already part of `ResourceCompatibility`.
- **Texture boundary.** The material table cannot vary sampled texture resources inside one draw. Texture handles and texture-present pipeline/bind-group requirements stay in the batch key, so a texture-backed record forms its own batch while the material table continues to hold only scalar/vector values. Diegetic fills are solid SDF surfaces; image leaves are not batched by this plan.
  - **Frame-local slot identity.** Each rendered material role receives a `MaterialSlotId` for the current frame's dense table. `MaterialSlotId` is a row index used by GPU records and is not durable across frames in the active plan.
  - **Repo-owned scalar/vector slot values.** The frame table storage layout is `MaterialSlotValues` plus a WGSL mirror, not raw `StandardMaterialUniform`. It may contain only fields that can vary per slot inside one draw. Texture presence, alpha mode, shader-def-driving flags, culling/sidedness, unlit/lighting mode, and shadow/prepass participation remain batch-key or pipeline splitters, with debug assertions that all records in a batch agree on them.
  - **Resource-only batch keys.** SDF, text, and panel-shape batch keys carry `pipeline_compatibility: PipelineCompatibility` and `resource_compatibility: ResourceCompatibility` directly, with scalar/vector PBR fields unrepresentable. These shared compatibility types contain only shared material-derived splitters. `PipelineCompatibility` contains material-driven pipeline/specialization facts such as alpha mode, shader defs, lighting mode, `double_sided`, and `cull_mode`. Reuse the existing `BatchAlphaMode` representation for hashable alpha mode; do not add a second alpha-key enum. Do not collapse facing compatibility into the current two-value `Sidedness` enum: `Sidedness` remains an authoring/cascade convenience that maps to `double_sided` and `cull_mode`, while direct `StandardMaterial` handles can still express `cull_mode = Some(Face::Front)`, `Some(Face::Back)`, or `None`. `ResourceCompatibility` contains material texture/sampler/bind-group resources. Renderer-specific splitters stay as direct fields on `SdfBatchKey` or `PathBatchKey` with doc comments explaining why they split that renderer. Use constructor-only APIs and tests so scalar/vector PBR fields cannot enter any batch key. `VisualBatchKey`, `InternedMaterialKey`, `VisualMaterialInterner`, and `BaseMaterialId` must stay absent from SDF and analytic keys.
  - **Batch key/resource/record relationship.** `SdfBatchKey` is the key in the SDF batch store, for example `HashMap<SdfBatchKey, SdfBatchResources>`. `SdfBatchResources` owns or tracks the batch entity, `SdfExtendedMaterial`, `SdfRenderRecord` buffer handle, capacity, and current record count for one compatible SDF batch. `PathBatchKey` is the key in the analytic-path batch store, for example `HashMap<PathBatchKey, PathBatchResources>`. `PathBatchResources` owns or tracks the batch entity, `PathExtendedMaterial`, `PathRenderRecord` buffer handle, related path/quad buffers, capacity, and current record count for one compatible text/shape path batch.
  - **Draw-order projection split.** `DrawOrder` derives `ScreenDepthBias` from `DrawZIndexRank`, while `ClipDepthNudge` and `OitDepthOffset` come from `DrawOrderIndex`. Existing draw-order tests must keep that split explicit.
  - **Batch sort fields.** A batch is one `Transparent3d` item, so its key must include the fields needed to keep Bevy's item sorting and this renderer's in-batch record order correct, such as z-index, render layers/view scope, and `ContiguousDrawnRun` when needed to preserve interleaving. These fields live directly on `SdfBatchKey` or `PathBatchKey`; do not introduce separate `SdfScreenBatchKey` or `PathScreenBatchKey` wrapper types unless implementation proves they remove real duplication.
  - **Draw-order diagnostics.** Current code derives batch material depth bias from the batch key's `DrawZIndexRank` and derives per-record shader values from each command's `DrawOrderIndex`. There is no active command-count warning; `depth_bias_stress.rs` is the visual probe for hardware depth-bias regressions.

## Phases

### Phase 0 — Rename Render Materials And Delete SDF Primitive Route · status: done

#### Work Order

**Goal:** Mechanical render-material renames and the SDF primitive/callout cleanup land before batching work, so later implementation phases start from a codebase where SDF means panel fills/borders only.

**Spec:**
- Start implementation by handing these exact renames to the user/editor as a mechanical rename pass:
  - `SdfPanelMaterial` -> `LegacySdfExtendedMaterial`;
  - `SdfPanelExtension` -> `LegacySdfExtension`;
  - `PathMaterial` -> `PathExtendedMaterial`;
  - current analytic-path `BatchKey` -> `PathBatchKey`;
  - current analytic-path `BatchGpu` -> `PathBatchResources`.
- Keep `PathExtension` unchanged. It is already the shared analytic-path extension for text and panel shapes.
- Do not rename `ShapeBatchKey` mechanically in Phase 0 unless doing so is no-behavior-change and low-risk. The desired final key name for the shared analytic path renderer is `PathBatchKey`; if `ShapeBatchKey` remains during the migration, Phase 6 must either remove it, fold it into `PathBatchKey`, or document a deliberately narrower remaining role.
- Do not rename the source-material cascade attributes in this phase. `SdfMaterial`, `TextMaterial`, and `ShapeMaterial` are introduced later as `Handle<StandardMaterial>` cascade values and are intentionally distinct from `SdfExtendedMaterial`, `PathExtendedMaterial`, and the migration-only `LegacySdfExtendedMaterial`.
- Delete the standalone/callout SDF primitive route **entirely** in this phase (decision D1: full deletion, no analytic adapter, no migration-with-preservation):
  - remove the Rust-side callout primitive *selector*: the `SdfPrimitiveKind` enum, `sdf_primitive_material`, and `SdfPrimitiveMaterialInput`. `sdf_panel_material` builds the rounded-rect material directly. The panel `SdfPanelUniform.sdf_kind`/`sdf_params` slots and the `sdf_panel.wgsl` kind branches stay (panels carry `0`/zero) until Phase 3 rewrites the panel SDF shader;
  - remove callout use of `LegacySdfExtendedMaterial` and `sdf_panel.wgsl`;
  - do **not** build a throwaway analytic callout adapter and do **not** map callout caps/segments to analytic closed forms. Callout capability is reintroduced later on the **PanelShape** path (Phase 9+), not analytic-path-direct;
  - remove the public `CalloutLine` component (and `CalloutVisual`). `CalloutCap` and its resolver (`callouts/caps.rs`) are **retained** as the panel-line cap API — `layout::line` consumes `CalloutCap`/`ResolvedCalloutCap`/`CalloutCapPrimitiveKind` and emits panel-shape primitives, a path independent of the deleted callout renderer. Do not preserve `CalloutLine` through an adapter;
  - delete the `update_callout_lines` behavior and `CalloutVisual` child lifecycle rather than preserving it;
  - update or remove every callout example, test, and probe that would otherwise break — the only invariant is a green build each phase;

**What worked:** Field-approval gate landed as an exhaustive no-`..` `StandardMaterialUniform` destructure in `MaterialSlotValues::from_standard_material_uniform` — a new upstream field breaks the build until classified. `MaterialSlotId`/`GpuMaterialSlotId`/`SdfPaintMaterial` boundary with a single `to_gpu`/`from_gpu` home + round-trip + sentinel-uniqueness tests. `PipelineCompatibility`/`ResourceCompatibility` carry no scalar/vector PBR values; full `AlphaMode` × `cull_mode` coverage. **What deviated from the plan:** Round-1 implementation shipped five real defects that the green gate did **not** catch (none observable until a shader binds the table): (1) rebind iterated *all* `PathExtendedMaterial`s every frame → per-frame asset churn; (2) the table upload was wired `RenderSystems::PrepareResources.before(PrepareAssets)`, an impossible constraint (`PrepareResources` ⊂ `Prepare`, chained after `PrepareAssets`); (3) the `BatchResourcesReady` boundary was hollow — no existing batch system was ordered into it; (4) the device row cap was unwired (defaulted to ~4 B); (5) parallax/depth-map classification dropped the parallax scalars while keeping `depth_map`, a latent batch-collision. **Surprises:** Bevy resolved to **0.19.0 final** (not rc.2); single `bevy_material` version, so `OpaqueRendererMethod` reaches via `bevy::material` re-export (the briefly-added direct dep was dropped). ### Retrospective

**What worked:** Clean typed foundation — `SdfRenderRecord` (160 B, indices-only), `GpuMaterialSlotId`-typed slots, `SdfPaintMask`, `SdfBatchKey`/`SdfBatchStore`, the atomic preflight, `assign_contiguous_runs` (R9), the R14 bounds recipe, and the S-B `sdf_batch_alpha_mode` mapping. The system pipeline wires through the Phase 2 schedule sets and relies on Bevy's default auto-sync-point so `register_sdf_batch_materials` sees same-frame reconcile spawns. Vertex-pulled + prepass shaders preserve rounded-rect/border/clip/subpixel/OIT behavior. Final gate green (540 tests).

### Phase 5 Review

Architect re-evaluation of remaining phases surfaced two judgment calls (type-enforcement aggressiveness; shadow semantics for newly-allowed SDF alpha modes). User decisions, recorded one at a time:

- **Adopt the `GpuMaterialSlotId` newtype.** Wrap the material-table row index in a distinct type at the GPU-record boundary instead of a bare `u32`, so the compiler rejects any other integer (paint mask, count) assigned where a slot id is expected. - **Reject the threaded three-state material status (`ResolutionStatus{Ready/Loading/Missing}`).** Do not thread a load-state enum through `CascadeProperty` and every material consumer. Revisit only if materials begin loading lazily/at runtime.
- **Reject the `SeededDefault` marker trait.** Do not add a build/startup check that every cascade default is seeded. Material defaults derive from `StandardMaterial`'s `Default` impl (Bevy-provided), so the unseeded-default failure path is not real here.

- **Adopt the field-approval gate as an exhaustive `StandardMaterial` destructure (no macro).** Rewrite the two `From<&StandardMaterial>` impls in `batch_key.rs` (`PipelineCompatibility` line 217, `ResourceCompatibility`) from field-access reads to a single `let StandardMaterial { ..all fields.. } = material;` with **no `..`**: splitters read into the key, table values / textures bound to their destination, everything else `field: _` with a one-word role comment. A Bevy upgrade that adds a `StandardMaterial` field then fails to compile at that one spot, forcing classification. **Zero-cost abstraction:** the destructure is a compile-time pattern only — `field: _` bindings emit no code, used fields compile to the same loads as today's `material.alpha_mode` access, so codegen is identical to the current form despite this running on the warm batch-routing path. The entire cost is the compile-time exhaustiveness check plus the source listing. Owning phase: whichever phase builds/owns the material-compatibility key (Phase 6/7).

The empty handle reaches a producer only in an unseeded (misconfigured) app, where the family renders nothing — no panic (the rejected `expect_seeded_default()` behavior). Doc corrected (the earlier "valid generic material" wording was wrong).

### Retrospective

**What worked:**
- The box-UV texture-sampling path, `gating_eq` inversion (color/material handle are now table-row inputs, not glyph-rebuild inputs), and the Phase 9 shape-type pre-staging (`PanelShape`/`PanelShapeOf`/`PanelShapes`, `PanelShapeMaterialSourceKey`) all landed; build/clippy/`nextest` green (577 tests).
- Splitting per-run material values into table rows while keeping only resource/pipeline facts in the batch key held: same-texture runs share one batch and write distinct box UVs (`same_texture_text_materials_share_texture_batch_and_write_box_uvs`).

**What deviated from the plan:**
- Four review findings were fixed by Claude directly (not codex) at the user's instruction, and a fifth defect — shadow receiving — was found outside the review and fixed with a built repro example:
  - **Shadow receiving (new, not in the review):** the batched vertex stage parks each quad's record index in `world_position.w`; Bevy's directional/spot shadow sampling multiplies the *full* `world_position` vec4 by the light's `clip_from_world`, so a non-1.0 `.w` displaces the shadow-map lookup per glyph. Fix: reset `pbr_input.world_position.w = 1.0` in `analytic_path.wgsl` after building the batched `pbr_input` (index recovery reads the raw `in.world_position.w`, unaffected). Visible repro: `examples/shadow_receiver_repro.rs` (flat text under a directional shadow band). No headless guard exists — shadow receiving needs a real GPU; the example is the proof.
  - **Normal-map text:** glyph quads carry UVs but no tangents, so a text material with `normal_map_texture`/`depth_map` would light from an undefined tangent basis. Text now strips those two channels before building its material candidate (`strip_tangent_dependent_maps`); UV-sampled channels survive.
  - **Material-only edit re-deriving glyph records:** a color/render-mode edit now routes to an in-place record refresh (`PathBatch::update_run_record`) that reuses the existing glyph quads. The GPU glyph-buffer re-upload was *already* prevented by the same-count in-place diff in `update_run`; this removes the redundant CPU re-derivation and the spurious placement-dirty mark.
  - **Stale records on missing glyph:** a run that was already routed now drops its records when a glyph goes missing mid-rebuild, instead of rendering last frame's glyphs; unrouted runs still self-heal.
  - **Material handles "lose splitters":** refuted — alpha/unlit/sidedness are cascade-owned for text by design (`analytic_material_slot_candidate` overwrites them intentionally).

**Surprises:**
- The `world_position.w` index-parking trick (which frees the UV varyings for material box UVs) silently breaks shadow *receiving* but not shadow *casting* (casting uses `world_position.xyz`). The defect is invisible until text sits inside a directional/spot shadow.
- The dual review tagged all four findings "blocker," but inspection downgraded most: #1 was a non-bug, #3's headline "re-uploads glyph buffer" was already prevented by the in-place diff (CPU-only residual), and #2 was latent (no current text authors normal maps).

**Implications for remaining phases:**
- Panel shapes (Phase 9) use the **same** `analytic_path` fragment shader, the same `PathQuadRecord` box-UV layout, and the same `world_position.w` index parking. The `world_position.w = 1.0` shadow fix is shared and already covers shapes — Phase 9 must not reintroduce a per-record `.w` write that defeats it.
- The tangent-strip is **text-only**. Shapes have a definable silhouette tangent basis (unlike glyphs), so Phase 9 must decide explicitly whether shapes support `normal_map`/parallax (real tangents) or strip them like text — the Phase 9 Work Order's texture-sampling spec does not currently address this.
- Phase 9's required "per-record diff path that compares silhouette separately from material slot updates" now has a text analog to mirror: the `render_only` flag (`PreparedPanelText`) + `update_run_record` (record/material refresh without geometry rebuild).
- `PathQuadRecord` is now 64 bytes (box-UV fields added); the shape box UV reuses that existing field rather than changing `PathRenderRecord`'s stride.

## Appendix B — Extension Discussion: Texture-Array Batching

Not part of the active plan. Do not implement this until a measured texture-heavy workload shows that texture splitting is the draw-call bottleneck. A few textured panels should stay on the existing texture-as-splitter path; this appendix is the escalation path for scenes with many distinct panel/run/shape textures.

**Problem this addresses.** The active plan classifies texture handles and texture-present requirements as `ResourceCompatibility` splitters (see the "Texture boundary" invariant): two records sampling different textures form different batches. That is correct for a workload where most fills/runs/shapes are solid color and textured records are the minority. It does **not** scale to a texture-heavy authoring goal — many panels each carrying distinct artistic texture maps. At, say, 1000 panels with per-panel-unique texture sets, the texture splitter dominates and batching collapses toward one draw per panel; the material table batches scalar/vector values but cannot merge across distinct textures. This appendix is the planned answer to that case.

**Two separable problems — do not conflate them.**
1. **Draw-call count.** Solved by binding a shared pool of textures once and indexing per-record, the same mechanism the material table already uses for scalar/vector values. This is what this appendix covers.
2. **Resident texture memory.** Binding-array indexing does not reduce VRAM: N distinct full-resolution maps resident at once is an N×size memory cost regardless of how they are bound. A texture-heavy goal needs streaming, atlas pages, mip/format compression, or virtual texturing as a *separate* effort. This appendix does not solve residency.

**Platform fit.** wgpu's Metal backend supports **texture** binding arrays but not general **buffer** binding arrays. That is enough for this appendix because it only moves texture handles out of the batch key; the active renderer's material table, SDF records, path records, glyph instances, and run records stay repo-owned storage buffers. Bevy 0.19's bindless support makes the same distinction: materials that use only textures/samplers/data buffers can use bindless without `BUFFER_BINDING_ARRAY`, but arbitrary bindless buffers are not available. Texture-array batching therefore targets the supported half of Metal's partial-bindless capability, not a fully bindless renderer.

**Fallback requirement.** Binding arrays are not universally available. The fallback for unsupported devices is today's correct-but-less-batched behavior: keep texture handles in `ResourceCompatibility` and split batches by texture resource. The plan amendment must keep this fallback rather than making texture-array support mandatory. Known unavailable or unsafe cases from Bevy 0.19/wgpu checks include WebGPU/WebGL-style targets without binding arrays, missing `TEXTURE_BINDING_ARRAY`, missing `SAMPLED_TEXTURE_AND_STORAGE_BUFFER_ARRAY_NON_UNIFORM_INDEXING`, devices that only allow uniform indexing into binding arrays, too few texture bindings for the required arrays, `shader_format_glsl` builds where Naga's GLSL backend cannot compile binding arrays, and buggy mobile GPUs such as Adreno <= 610.

**Design direction (to be settled in a real amendment before coding).**
- Promote the texture set from a `ResourceCompatibility` batch-key splitter to a **table-style index**: the record carries a texture-pool index (per channel as needed), indexed exactly as `MaterialSlotId` indexes `MaterialSlotValues`. Records sharing pipeline compatibility but differing only in texture *index* then share a batch.
- Keep the existing box-UV + `uv_transform` plumbing (Phase 2 `compute_material_sampled_uv`, the per-record box UV in `SdfRenderRecord` / per-glyph instance record / `PathRenderRecord`): it already supplies the per-record sampling coordinate a texture-array read needs. No new UV concept is required.
- A record still splits batches on facts a texture *array* cannot vary within one draw: differing sampler address/filter modes, differing texture formats or array layer dimensions where the pool is homogeneous, and any mesh-attribute requirement (tangents for `normal_map_texture`). Those stay pipeline/resource splitters.

**Bevy `#[bindless]` vs. a repo-owned texture array.** This appendix is about record-level texture batching, not ordinary mesh/material batching. Bevy `#[bindless]` is designed around many material assets allocated into bindless slabs; its shaders fetch a material slot from Bevy's mesh uniform (`material_and_lightmap_bind_group_slot`) and use that slot to index the material's texture/sampler resources. Diegetic batching is shaped differently: one `SdfExtendedMaterial` or `PathExtendedMaterial` batch asset draws many SDF/text/shape records, and each record may need a different texture. The texture index therefore belongs in our record/material-table data, not in the one Bevy material slot for the whole batch entity.

Preferred route: a repo-owned texture binding array indexed by a record field. This mirrors the material table: build a per-frame texture pool, bind its texture/sampler arrays once, and store per-channel texture indices in `MaterialSlotValues` or an adjacent texture-table row selected by the record. This keeps the pool, growth/rebind discipline, fallback branch, and bind-group layout under this renderer's control.

Use Bevy `#[bindless]` only if a prototype proves it can expose the exact per-record texture indices this renderer needs without forcing one Bevy material allocation per record or fighting the existing manual bindings. Until such a prototype exists, treat `#[bindless]` as evidence that texture/sampler binding arrays are viable, not as the chosen implementation for diegetic record-level texture batching.

**Height/depth maps are in scope of this extension.** `depth_map` (the parallax/relief height texture) is a `StandardMaterial` texture channel like any other, so it is subject to exactly this texture-array batching: many *distinct* height maps split batches today and would share a draw only once promoted into the texture pool. If this appendix is implemented, the height map must join the pool alongside the base-color/emissive/metallic-roughness/normal/occlusion channels, and the parallax scalar table values (see [Tangent-Bitangent-Normal For Analytic Materials](tangent-bitangent-normal.md)) index the same per-record row. Parallax *shader* support lives in the TBN design doc and is independent of this batching work — Appendix B makes many distinct height maps share a draw; the TBN shader path makes the shader actually displace UVs from one.

**Prerequisite from the active plan.** This extension reuses, and does not modify, the Phase 2 substrate: the u32-indexed table model, the per-record box UV, and `uv_transform` as a non-splitting table value. The active plan's "textures split batches" rule is the v1 simplification this appendix later relaxes; it is not a permanent ceiling. When implemented, write a plan amendment stating the measured draw-call problem the texture splitter causes, the feature-detection and fallback behavior, the pool-growth/rebind model, the homogeneity constraints on the array, the depth_map channel handling, and why the chosen route can supply per-record texture indices without changing the batch architecture.
