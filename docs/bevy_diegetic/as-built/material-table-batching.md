# Material Table Batching

## What it is

Diegetic UI panels render three families of translucent surface — SDF panel fills/borders, analytic-path text, and analytic panel shapes — as batched draws on Bevy 0.19 / wgpu 29, with OIT (`StableTransparency`) resolving composition. The batching problem is that any per-surface difference in a `StandardMaterial`'s scalar/vector PBR values (base color incl. alpha, emissive, metallic, roughness, reflectance, transmission, thickness, attenuation, clearcoat, anisotropy, `ior`, `uv_transform`) would normally force a separate draw. The fix: those values are **material-as-data**. Each frame the renderer builds one dense **material table** (`Vec<MaterialSlotValues>` uploaded to a shared storage buffer); every GPU record carries only a `u32` row index into it. Batch keys carry *no* scalar/vector values — only pipeline/resource compatibility (alpha mode, textures, sidedness, cull mode, lighting, shadow/prepass participation) plus sort/order fields. Two fills / text runs / shape primitives that differ only in table values share one batch and read different rows; they split only when a real pipeline or resource fact differs. The table is rebuilt from scratch each frame (frame-local slot ids), so animating material values is a row rewrite, never a re-batch.

## How it works

### The three render families and their record/batch types

| Family | Record (GPU) | Batch key | Batch store / resources | Render material |
|---|---|---|---|---|
| SDF fill/border | `SdfRenderRecord` (160 B) + `SdfMeshRecord` (16 B) | `SdfBatchKey` | `SdfBatchStore` → `SdfBatch` / `SdfBatchResources` | `SdfExtendedMaterial = ExtendedMaterial<StandardMaterial, SdfExtension>` |
| Text | `PathRenderRecord` (96 B) + `PathQuadRecord` (64 B) + `PackedPathRecord` (48 B) | `PathBatchKey` | `PathBatchStore` → `PathBatch` / `PathBatchResources` | `PathExtendedMaterial = ExtendedMaterial<StandardMaterial, PathExtension>` |
| Panel shapes | same analytic records as text | `PathBatchKey` | same `PathBatchStore` machinery | same `PathExtendedMaterial` |

Text and shapes **share** `PathExtendedMaterial` / `PathExtension` and the same `analytic_path` shader — there is no `TextExtension`/`ShapeExtension`. `PathExtension` (`analytic_paths/material.rs`) owns the analytic storage-buffer plumbing at bindings 100–105, plus `RenderMode`, AA, and `vertex_pull` specialization via `PathExtensionKey`. `SdfExtension` (`fill_batch.rs`) owns the `SdfRenderRecord` buffer + inert mesh at bindings 107/108, and specializes on `SdfPipelineMode { VertexPulled, MeshAttributes }` via `SdfExtensionKey` (`MeshAttributes` exists but is test-only today). Both extensions bind the shared material table at `MATERIAL_TABLE_BINDING = 106`.

`DrawBatchFamily { SdfSurface, PanelShape, Text }` and `DrawSortTier { Surface, PanelShape, Text }` are derived from `RenderCommandKind` (`layout/render.rs`: `draw_batch_family()` / `draw_sort_tier()`), keeping the coarse cross-family ordering.

### The frame material table (`render/material_table.rs`)

- **`MaterialSlotValues`** — the repo-owned row: `base_color`, `emissive` (`.w` = exposure weight), `attenuation_color`, `uv_transform: Mat3`, `reflectance: Vec3`, `roughness`, `metallic`, `diffuse/specular_transmission`, `thickness`, `ior`, `attenuation_distance`, `clearcoat`, `clearcoat_perceptual_roughness`, `anisotropy_strength`, `anisotropy_rotation: Vec2`. Built by `from_standard_material_uniform(StandardMaterialUniform)` which exhaustively destructures the uniform (no `..`) — the field-approval gate. A WGSL mirror lives in `material_table.wgsl`.
- **`MaterialSlotId`** — CPU-side, frame-local, valid-only row index. `TryFrom<u32>` rejects `u32::MAX` and out-of-range. **`GpuMaterialSlotId`** — the opaque `u32` written into GPU records; `From<MaterialSlotId>`; the `INVALID_GPU_MATERIAL_SLOT = u32::MAX` sentinel is only reachable via `SdfPaintMaterial::to_gpu`/`from_gpu`. **`SdfPaintMaterial { Authored(MaterialSlotId), NotAuthored }`** is the CPU presence enum for a fill/border role and the single CPU↔GPU sentinel-conversion home.
- **Projection contract**: `MaterialSlotCandidate { values, pipeline_compatibility, resource_compatibility }`, the private `MaterialSlotInput` trait (`type Key`, `key()`, `material_slot_candidate()`), and `append_material_slot(builder, input) -> MaterialSlotAppend { Appended(MaterialSlotAppended<K>), DroppedLimit }`. Implemented by `SdfMaterialSlotInput`, `PanelShapeMaterialSlotInput`, and (text) the borrow-and-project function `text_material_candidate_for_frame` + `analytic_material_slot_candidate`.
- **Ownership/lifecycle**: one main-world `FrameMaterialTableBuild` owns the single `FrameMaterialTableBuilder`. Producers append in a fixed serialized order (SDF → text → shape) for deterministic row order. Device row cap comes from `RenderDevice::limits().max_storage_buffer_binding_size`, wired in `MaterialTablePlugin::finish()`.
- **Schedule** (named `PostUpdate` sets, `MaterialTablePlugin`):
  1. `MaterialTableAppendReady` (`.before(TransformSystems::Propagate)`) — `clear_frame_material_table`; producers resolve materials and append rows **and** build records here.
  2. post-`Propagate` — world transforms, sort centers, batch placement, hand-written `Aabb` (before `VisibilitySystems::CheckVisibility`).
  3. `BatchResourcesReady` (`.after(Propagate)`) — all SDF/text/shape batch-material create/grow/register/unregister finishes. `register_path_batch_materials::<DiegeticTextBatch>` / `<DiegeticPanelShapeBatch>` and `register_sdf_batch_materials::<DiegeticSdfFillBatch>` run here into `BatchMaterialTableRegistry`.
  4. `MaterialTableUpdatedToCurrent` (`.after(BatchResourcesReady)`, last in `PostUpdate`) — chained: freeze (append-after-freeze panics) → `ensure_material_table_buffer_handle` → `rebind_registered_material_table_buffers` → `update_material_table_buffer_data` (main-world `ShaderBuffer::set_data` padded to capacity, mirrors `analytic_paths/atlas.rs`) → warn on `DroppedLimit`.
  5. Extract — `extract_frame_material_table` produces read-only `ExtractedFrameMaterialTable { table, handle }`. **There is no render-world upload system** (the planned `prepare_material_table_buffer`/`PrepareResources` path was unschedulable). Records and rows extract together each frame, so a render-world record never indexes another frame's table.

The rebind pass rewrites registered `SdfExtendedMaterial` / `PathExtendedMaterial` asset handles so extracted materials already point at the current table buffer *before* material bind-group prepare. Rebind is registry-only and skips when the handle is unchanged (`RegisteredSdfMaterial`/`RegisteredPathMaterial { bound_handle }` + `pending_rebind`). `purge_stale_registered_material_table_buffers` drops dead-entity entries each frame; `unregister_path` is the eager handle-replacement path.

### Batch keys and compatibility (`render/batch_key.rs`)

`SdfBatchKey` and `PathBatchKey` both carry `pipeline_compatibility: PipelineCompatibility` and `resource_compatibility: ResourceCompatibility` directly, plus renderer-agnostic sort fields (`z_index`, `z_index_rank`, `layers: BatchRenderLayers`, `shadow: VisualShadow`, `contiguous_drawn_run: ContiguousDrawnRun`) and family. Scalar/vector PBR values are structurally unrepresentable in either.
- `PipelineCompatibility` (`From<&StandardMaterial>` via the exhaustive destructure): `alpha: BatchAlphaMode`, `double_sided`, `cull_mode: Option<Face>`, `unlit`, `fog_enabled`, `opaque_render_method: BatchOpaqueRendererMethod`, `deferred_lighting_pass_id`.
- `ResourceCompatibility`: per-channel texture handles + `BatchUvChannel` selectors + `flip_normal_map_y` (base color, emissive, metallic-roughness, normal, occlusion). `apply_resource_compatibility_to_standard_material(base, compat)` is the **only** place texture handles move from a key back into a created batch material; `apply_pipeline_compatibility_to_standard_material` applies the non-texture pipeline facts.
- `BatchAlphaMode` is the one hashable alpha representation (reused, not duplicated). `VisualShadow` unifies `ShadowCasting` / `SurfaceShadow` / `GlyphShadowMode`.

### Cascade material-source model (`render/material.rs`, `cascade/`)

`CascadeProperty` is `Clone + PartialEq` (generalized from `Copy`). Three source-handle attributes cascade: `SdfMaterial(Handle<StandardMaterial>)` (element surfaces / fills / borders), `TextMaterial(...)`, `ShapeMaterial(...)`, each with `override_*_material` / `inherit_*_material` / `resolved_*_material`. These are **source handles**, not render materials. Each producer reads `Resolved<SdfMaterial>` / `Resolved<TextMaterial>` / `Resolved<ShapeMaterial>` at the surface/run/shape node, then projects the asset behind the handle through the single helper `material_asset_for_frame(materials, asset_server, handle, default_handle)`, which applies the readiness gate (path-backed handle still `Loading` → `None` → hold/skip; missing/failed → seeded default). Authored per-surface color / `TextStyle::with_color` / `PanelLine::color` etc. overwrite the projected row `base_color` before projection; they do not split batches. `CascadeDefault<{Sdf,Text,Shape}Material>` is seeded from `default_panel_material()` in `RenderPlugin` via `seed_default_material_cascades`; the `Default` impl returns an empty `Handle::default()` only to satisfy the generic `init_resource` bound and never renders in a seeded app.

### Draw order and depth (`render/draw_order.rs`)

Diegetic draw order owns depth; `StandardMaterial::depth_bias` is ignored as source data. `DrawZIndex` (authored) → `DrawZIndexRank` (dense rank of distinct z-indices in a panel) → `ScreenDepthBias` (Bevy hardware integer depth-bias, advances once per z-index band, set on the batch material). `DrawOrderIndex` (dense rank after sorting `(DrawZIndex, DrawSortTier, CommandIndex)`) → `ClipDepthNudge` (non-OIT shader nudge) and `OitDepthOffset` (per-record OIT z offset, `OIT_DEPTH_STEP`). `DrawCommandDepth` caches all of these per command. Ordinal newtypes (`CommandIndex`, `ElementIndex` with `CHILD_DIVIDER` sentinel, `ShapeOrdinal`, `PrimitiveOrdinal`) share this one home.

### Data flow, end to end

Layout emits `RenderCommand`s → `panel_geometry.rs` resolves SDF surfaces into `ResolvedSdfSurfaceRegistry` (`upsert_panel`/`remove_panel`); text and shape producers resolve their sources. Each producer, in the append window, resolves its material handle → projects `MaterialSlotCandidate` → `append_material_slot` → stores the returned `MaterialSlotId` (as `GpuMaterialSlotId`) into its GPU record and routes the record to a batch by compatibility. Post-propagation systems compute world transforms and hand-written bounds. `BatchResourcesReady` builds/grows batch materials from key compatibility; rebind repoints table-buffer handles; extract carries records + rows together. Each family reads its row through a `pbr_input_from_material_table` helper that applies the guard order and feeds Bevy PBR lighting: the SDF path (`sdf_panel.wgsl`) imports it from `shaders/sdf_material_table.wgsl` (which carries the SDF-specific `SDF_STRIPPED_MATERIAL_GROUP` prepass branch); the analytic path (`analytic_path.wgsl`) imports the parallel helper from `render/material_table.wgsl`. The `MaterialSlotValues` WGSL mirror, `INVALID_GPU_MATERIAL_SLOT`, and `compute_material_sampled_uv(box_uv, uv_transform)` (the sampling coordinate for all texture channels) live in `render/material_table.wgsl`, shared by both.

## Invariants

Preserve these or something breaks:

1. **Material-as-data / scalar-free keys.** No scalar or vector PBR value may enter `SdfBatchKey`, `PathBatchKey`, `PipelineCompatibility`, `ResourceCompatibility`, or the shape merge key's value store. Keys carry compatibility + sort fields only. The compile/test guards (exhaustive destructure, no-scalar assertions, merge-key guard) exist to keep this true; do not add `..`.
2. **Exhaustive-destructure field-approval gate.** `PipelineCompatibility::from` / `ResourceCompatibility::from` (`batch_key.rs`) and `MaterialSlotValues::from_standard_material_uniform` destructure `StandardMaterial` / `StandardMaterialUniform` with **no `..`**. A new upstream Bevy field is an `E0027` compile error until classified in place (table / pipeline / resource / draw-order / unsupported / deferred) with a `_<name>_<class>` binding. This is zero runtime cost.
3. **One implementation target.** No SDF-local material interner, no record-local SDF color, no `VisualMaterialInterner` / `VisualBatchKey` / `InternedMaterialKey` / scalar-value `BaseMaterialId`. All three families use the frame table exclusively. The legacy per-surface SDF route (`PanelSdfMesh` / `PanelSdfSurface` / `spawn_sdf_quad` / `LegacySdfExtendedMaterial`) is gone; `PanelInteractionMesh` is picking-only. `sdf_panel.wgsl` is the batched SDF path's live production shader (all stages, via `EMBEDDED_SDF_PANEL_BATCH_SHADER_PATH`), importing coverage from `sdf_stroke.wgsl`.
4. **Texture boundary.** The table holds only scalar/vector values. Texture handles and texture-present pipeline/bind-group requirements stay in `ResourceCompatibility` (batch key), so a texture-backed record forms its own batch. `apply_resource_compatibility_to_standard_material` is the sole path from key → batch-material texture handles.
5. **Depth owned by draw order.** `DrawZIndexRank`→`ScreenDepthBias` (per-band hardware bias on the batch material), `DrawOrderIndex`→`ClipDepthNudge`/`OitDepthOffset` (per-record). This split is explicit and tested; do not collapse it or route depth through `StandardMaterial::depth_bias`.
6. **Frame-local slot identity.** `MaterialSlotId` is a row index valid only for the current frame's dense table. Every live record must re-append/refresh its row and rewrite its `GpuMaterialSlotId` every frame, even when geometry/placement/atlas are unchanged. Changed-only gates may skip geometry work but never the material row/id refresh.
7. **Material-handle readiness is a use-site gate.** `material_asset_for_frame` holds/skips a producer entry when its asset is still `Loading`; missing/failed falls back to the seeded default row. Never reuse stale table values across a handle→invalid→valid transition. No threaded three-state resolution enum.
8. **Cascade defaults via `init_resource` + `RenderPlugin` seeding.** No `SeededDefault` marker trait, no `expect_seeded_default()` panic. `SdfMaterial`/`TextMaterial`/`ShapeMaterial` must not derive `Default`; the empty-handle `Default` lives only on `CascadeDefault<_>`. Cascade attributes must stay cheap to clone (`Handle`, not owned `StandardMaterial`); a `size_of` budget asserts it.
9. **Layout-neutral material swaps need a change trigger.** A producer that reads a material `Resolved<_>` must also wake on `Changed<Resolved<thatMaterial>>` (text: yes; SDF `build_panel_geometry`: `Or<(Changed<ComputedDiegeticPanel>, Changed<Resolved<SdfMaterial>>)>`; the shape producer runs unfiltered per frame — keep it unfiltered).
10. **ShaderBuffer rebind hazard.** `ShaderBuffer::set_data` with a changed byte length can recreate the wgpu buffer while old bind groups still point at the old one. Every record/table upload is padded to capacity; growth creates new buffer assets and rewrites material handles in the ordered rebind pass *before* extraction. Keep every batch entity's marker (`DiegeticTextBatch` / `DiegeticPanelShapeBatch` / `DiegeticSdfFillBatch`) + `MeshMaterial3d<…>` so registration keeps rebinding it — silent rebind loss is invisible to headless tests.
11. **Batch bounds are hand-written.** SDF/text/shape batch meshes are inert, so bounds must be hand-computed: union transformed `mesh_half_size` corners (AA padding already baked in) with per-record clip, batch center = union centroid, local `Aabb` = union − center, attach `NoAutoAabb`, schedule between Bevy bounds calc and `CheckVisibility`. `SdfRenderRecord::transform` (world-space) is the only transform the shader uses; the batch entity transform is for visibility/sort only.
12. **Atomic multi-role append.** An SDF fill+border pair appends both rows atomically: preflight required-role count against remaining capacity; if the border append returns `DroppedLimit` after the fill appended, roll back / skip the whole record. Never emit a record with one valid required slot + one sentinel. Increment the dropped-record counter and warn.
13. **Cross-batch interleave preserved.** Split otherwise-compatible records into maximal compatible runs in `DrawCommandDepth::draw_order_index()` order (`ContiguousDrawnRun` forces same-batch when equal). A batch is one `Transparent3d` item sorted by union centroid, so in-batch record order is the sole in-band compositing order.
14. **Prepass/shadow reads the retained table (one source of truth).** Keep the SDF material group alive in depth/prepass by mapping shadow-casting batches' GPU alpha mode to `MAY_DISCARD` (`Opaque → Mask(0.0)`); the authored mode stays in `PipelineCompatibility`. Prepass reads `fill_alpha` from the table and border presence from `border_material != INVALID_GPU_MATERIAL_SLOT`, guarding role-present → `!= INVALID` → `< arrayLength`. Denormalized prepass-only fields are a documented fallback only. A shader-def tripwire guards binding 106/107/108 presence through prepass so a Bevy material-group-stripping change forces review.
15. **All table reads go through `pbr_input_from_material_table`.** The read guard order (role-present → `!= INVALID_GPU_MATERIAL_SLOT` → `< arrayLength(&material_table)`) is mandatory; a source-scan tripwire rejects direct `material_table[...]` reads. The helper's `SDF_STRIPPED_MATERIAL_GROUP` branch is SDF-specific; analytic reuse must share/fork it deliberately.
16. **Typed dirty categories.** `MaterialDirty` / `PlacementDirty` / `GeometryDirty` are distinct structs (not a bitset). A material-only refresh is type-incapable of dirtying geometry/atlas.
17. **New-type comments required.** Every record/id/key/resources/extension/cascade-handle type and field carries a doc comment stating what it is and what it connects to.
18. **`SHADER_SIZE` asserts are load-bearing.** `SdfRenderRecord` = 160, `SdfMeshRecord` = 16, `PackedPathRecord` = 48, `PathQuadRecord` = 64, `PathRenderRecord` = 96, `CurveRecord` = 80. Any layout change re-asserts these, updates the WGSL mirror structs in lockstep, and refreshes `EXPECTED_SHADER_FNV1A` in `coverage_probe.rs` when shader text changes.

## Calibration / gotchas

- **Bevy 0.19.0 final** (not rc.2): a single `bevy_material` version; reach `OpaqueRendererMethod` via the `bevy::material` re-export, not a direct dep.
- **`SDF_AA_PADDING = 0.001`** and the `- pad`/`+ pad` clip expansion preserve anti-aliased edges at clip boundaries; AA padding is baked into `mesh_half_size`. `OIT_DEPTH_STEP = 0.000_001` (24-bit OIT depth packing) with an `OIT_MIN_DEPTH` floor so coplanar records fall back to list insertion order rather than vanishing.
- **`world_position.w` index parking breaks shadow *receiving*.** The batched vertex stage parks the record index in `world_position.w`; Bevy's directional/spot shadow sampling multiplies the full vec4 by `clip_from_world`, displacing the shadow-map lookup per glyph/record. Fix (shared by text and shapes): reset `pbr_input.world_position.w = 1.0` in `analytic_path.wgsl` after building the batched `pbr_input`. Shadow *casting* is unaffected (uses `.xyz`). No headless guard — verified by `examples/shadow_receiver_repro.rs`.
- **Vertex-pull correction:** subtract `mesh[instance_index].first_vertex_index` before deriving the local vertex/record index; collapse capacity-tail (padding) records before any table read.
- **Non-`Blend` alpha modes render hard edges by design.** Analytic coverage AA is straight alpha (`analytic_path.wgsl` `final_alpha = coverage * fill_color.a`), which composites smoothly only under `Blend`. `Premultiplied`/`Add`/`Multiply` text/shapes look hard-edged — this is authored behavior, not a batching/AA regression. Author texture-backed validation surfaces as `Blend` or label the expected edge.
- **Text strips `normal_map_texture` + `depth_map`** (`strip_tangent_dependent_maps`) before building its candidate — glyph quads carry no tangent basis. Shapes strip them too. Real analytic normal-map/parallax via a synthesized tangent basis is deferred to the TBN follow-on.
- **Parallax scalars physically deferred:** `parallax_depth_scale` / `max_parallax_layer_count` / `max_relief_mapping_search_steps` have a decided table classification but are dropped from `MaterialSlotValues` (bound as `_..._table_when_appendix_d`) because no diegetic shader applies parallax — two records differing only in parallax render identically today. `lightmap_exposure` is `_deferred_until_lightmap_design`. `depth_map` is already a live `ResourceCompatibility` splitter.
- **`MaterialSlotValues::from(&StandardMaterial)` allocates an empty `RenderAssets<GpuImage>` per call** — hoist it once outside per-record producer loops to avoid per-row churn.
- **Headless nextest is blind to render-world correctness.** Shaders compile lazily; schedule/upload/rebind/shadow correctness needs a real device. Keep the guards (schedule-resolution test, no-per-frame-asset-churn change-detection test, 106/107/108 binding tripwire) and repro examples; rasterized parity is a BRP/GPU harness job, permanently split from `cargo nextest`.
- **Per-panel batch/upload counts are unobservable** — batches merge scene-wide by compatibility key. `batch_validation` validates via a derived internal-consistency latch over live `DiegeticPerfStats` per family (per-batch rows == family batch count; per-batch records sum == family record total; no empty batch), not per-panel predictions. Per-panel readout is record count + material-slot count + share/split reason only.
- **Lighting/sidedness are panel-cascade-scoped, not per-surface.** There is no per-element/per-shape lighting/sidedness authoring surface; fills, shapes, and text all resolve these from the panel cascade via the forward `render::apply_sidedness`. The old backward inference (`shape_sidedness`/`shape_lighting` reading `unlit`/`cull_mode` off the resolved material) was deleted. Shape *alpha mode* does flow from the resolved material (`base.alpha_mode`) and does split batches.
- **`PanelShapeBaseColorIdentity::Source` keys on color only** (dropped the authoring entity), so same-color shapes from different sources share one table row.
- **`PathQuadRecord` is 64 B with box-UV fields** (`box_uv_min`/`box_uv_size`/`box_uv_flip_x`); the shape/text box UV rides there, not on `PathRenderRecord`. The `box_uv_flip_x` slot reserves the reverse-text follow-on (a per-glyph UV flip that does not split a batch).
- **Registry cap:** creating/despawning a batch each frame must not accumulate stale registry entries — `purge_stale_registered_material_table_buffers` bounds it (10-frame cap test).

## Why

- **Frame-built dense table, not a durable slot allocator.** Rebuilding rows + records together each frame makes a render-world record structurally unable to index a stale table (Bevy `extract` replaces resources per frame), so there are no tombstones, retirement queues, epochs, or generation checks. The durable `SharedMaterialTable` allocator was measured against and left as a documented alternative; the simple model won on complexity and satisfied the measured cost. Implement the allocator only with a written amendment citing a measured failure.
- **Cascade carries source *handles*, not owned materials.** A `Handle<StandardMaterial>` is cheap to clone (required by the `Clone` cascade) and lets edits to a shared asset update table rows without changing cascade resolution or batch identity — asset edit ≠ handle swap. Owning `StandardMaterial` in the cascade would make every value tweak a cascade invalidation and blow the clone budget.
- **`PathExtension` shared by text and shapes, not split.** Both are vertex-pulled analytic-path draws reading the same storage buffers and the same `analytic_path` shader; separate `TextExtension`/`ShapeExtension` would duplicate all plumbing for no pipeline difference. Split only if a proven pipeline requirement appears. The differences that do exist (tangent-strip, silhouette vs run box UV) are per-record data, not per-extension.
- **Depth owned by draw order, not `StandardMaterial::depth_bias`.** Diegetic ordering is authored (z-index bands + command order) and must survive batching, which merges surfaces from different source materials into one draw. Hardware bias per z-index *band* (via `DrawZIndexRank`) keeps the integer depth-bias path from advancing per command, while per-record OIT/clip offsets (via `DrawOrderIndex`) still separate commands inside a band. Source-material `depth_bias` would be both a scalar in a batch key (forbidden) and semantically wrong.
- **Exhaustive destructure gate.** `StandardMaterialUniform`/`StandardMaterial` are all-`pub` and not `#[non_exhaustive]`, so a `no-..` destructure is a free compile-time tripwire: a Bevy upgrade that adds a PBR field cannot silently land in the wrong bucket (a scalar leaking into a key = a batch-collision correctness bug, or a splitter dropped = wrong pixels). It costs nothing at runtime and forces a human classification at exactly one site.
- **Material identity stays a shape merge-key splitter.** One `PathRenderRecord` binds exactly one `MaterialSlotId`, and one material source maps to one merged render record, so material-source identity must gate record granularity just as alpha mode gates batch granularity. Only the scalar *value storage* moved to the table; an opaque non-value source-identity splitter remains so same-silhouette/different-material primitives keep distinct records/rows within one batch.
