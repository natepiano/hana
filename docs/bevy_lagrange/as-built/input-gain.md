# Device-specific camera input gain

## What it is

Cameras need per-device tuning: a trackpad, a mouse wheel, a pinch gesture, and a gamepad stick all deliver raw motion at wildly different scales, and users want to dial each one in without the app rewriting camera math. This feature puts that tuning in the **binding/preset layer** as an authored *input gain* multiplier applied to raw device input at capture, before it becomes a semantic orbit/pan/zoom action. The camera controller's own per-axis **sensitivity** (the master gain in `OrbitCam`/`FreeCam`) stays untouched and remains the final multiplier. The two concepts are deliberately named apart in the shipped code — `InputGain` scales the signal at the input binding; `Sensitivity` scales a camera axis's response downstream — so device calibration never leaks into camera operation code.

> Naming note: an earlier design called the binding-layer value "sensitivity." The shipped code uses **input gain** (`InputGain`, `OrbitCamInputGain`, `.with_input_gain(...)`, `mouse_input_gain(...)`) and reserves **`Sensitivity`** exclusively for the camera-axis master gain. Everything below uses the shipped names.

## How it works

The input stack is split into a shared layer (`crates/bevy_lagrange/src/input/`) and per-controller layers (`crates/bevy_lagrange/src/orbit_cam/input/` and `crates/bevy_lagrange/src/free_cam/input/`). Data flows binding → preset → adapter/native install → controller.

**1. Input-gain value types (`input/bindings/descriptor.rs`).**
`InputGain(pub f32)` is the authored per-binding multiplier: `DEFAULT = 1.0`, `DISABLED = 0.0`, `is_enabled()` returns `self.0 != 0.0`, and `validate()` accepts finite `>= 0.0` values and otherwise returns `BindingsError::InvalidScale`. Each flattened `InputBindingEntry` carries its own `input_gain: InputGain` stored separately from the signed scale. `InputBindingEntry::install_modifiers()` is the single place native gain and scale combine: it calls `InputBindingModifiers::with_input_gain(input_gain)`, which folds the gain into the modifier scale via `combined_scale(scale, input_gain)`. `InputBindingScale` remains a signed `Uniform(f32) | Axes2d(Vec2)` that encodes direction and axis mapping.

**2. Grouped and per-binding input gain.**
`OrbitCamInputGain` (`orbit_cam/input/bindings/input_gain.rs`) holds `orbit/pan/zoom: InputGain` with `new()`, `uniform(f32)`, fluent `.orbit/.pan/.zoom(f32)`, accessors `orbit_input_gain()`/`pan_input_gain()`/`zoom_input_gain()`, and `validate()`. `FreeCamInputGain` (`free_cam/input/bindings/input_gain.rs`) is the FreeCam analogue. Individual custom bindings get `.with_input_gain(f32)` (`orbit_cam/input/bindings/binding_kinds.rs`), which wraps the binding in `OrbitCamBindingWithInputGain<T>`, preserving forwarding setters so modifier-order ergonomics hold. Touch tuning lives in `OrbitCamTouchBindingConfig`, which carries per-action (orbit/pan/zoom) gain.

**3. Preset lowering (`orbit_cam/input/bindings/preset/`).**
`OrbitCamPreset` (`preset/enum_preset.rs`) is a payload-carrying enum — `SimpleMouse(OrbitCamSimpleMousePreset)`, `BlenderLike(...)`, `Keyboard`, `SimpleMouseKeyboard`, `BlenderLikeKeyboard`, `Gamepad(OrbitCamGamepadPreset)`. Callers use associated constructors (`OrbitCamPreset::blender_like()`, each `TypedPreset::default().into()` via `From`), never direct variant construction. `kind()` returns the setting-insensitive `OrbitCamPresetKind`; `name()` = `kind().name()`; `to_bindings()` lowers the full payload into validated `OrbitCamBindings`. Source setters come from the traits in `input/source_input_gain.rs` — `MouseInputGain`/`SmoothScrollInputGain`/`GamepadInputGain` (each with `type Gain` and `mouse_input_gain`/`smooth_scroll_input_gain`/`gamepad_input_gain`) — plus inherent `const fn` forwarders on the concrete pointer/gamepad presets so the advertised snippets compile without trait imports. Composed presets forward pointer setters into their pointer child. Lowering **stamps** authored `InputGain` onto generated binding/config payloads; it must not pre-multiply `InputBindingScale`, since `install_modifiers()` is the sole native composition point. `FreeCamPreset` mirrors this shape in `free_cam/input/bindings/preset.rs`. `OrbitCam::with_preset(impl Into<OrbitCamPreset>)` and `OrbitCamInputMode::with_preset(...)` are the install entry points.

**4. Adapter install / inject / resolve + smooth scroll (`orbit_cam/input/adapter/`).**
The adapter stages raw Bevy input into `adapter_*` actions before BEI resolves semantic intent: wheel line scroll → `adapter_zoom_coarse`; pixel scroll → `adapter_orbit`/`adapter_pan`/`adapter_zoom_smooth`; pinch/touch/button-drag → smooth-zoom units. Each source's gain is applied at the point its contribution is known (in `inject.rs`), not deferred to collection. Smooth scroll is a hybrid adapter/BEI path with **exactly one scaling owner**: `selected_trackpad_binding()` (`adapter/inject.rs`) chooses one concrete enabled binding via `TrackpadScrollCandidate { target, index, mod_keys, input_gain }`, ranked by `(mod-key count, target priority orbit<pan<zoom, authored index)`; `refresh_trackpad_conditions()` activates the matching `TrackpadBindingCondition { target, index, mod_keys, active }` (`adapter/install.rs`); and the installed `Scale` is `Scale::splat(input_gain.value())` for orbit/pan and `PIXEL_SCROLL_SCALE * input_gain.value()` (with `SwizzleAxis::YXZ` and the existing zoom negation) for zoom. Injection keeps raw pixel scroll raw and resolution does not re-scale, so the gain is applied once. Disabled (`0.0`) entries are filtered by `enabled_trackpad_orbit/pan/zoom` and the other `enabled_*` accessors before selection, install, source flags, and pinch suppression. `OrbitCamAdapterFrameSources` records per-frame source attribution; tuned wheel zoom still reports `WHEEL`, button-drag still `MOUSE`, pixel scroll still `SMOOTH_SCROLL`. Normalization constants (`PIXEL_SCROLL_SCALE`, `PINCH_GESTURE_AMPLIFICATION`, `TOUCH_PINCH_SCALE`, `BUTTON_ZOOM_SCALE`) remain source-unit conversions that gain multiplies, not replaces.

**5. Lifecycle / debounce (`input/lifecycle.rs`, `input/mode_reconciliation.rs`, `input/modes.rs`).**
Mode reconciliation lowers and validates preset/draft payloads into resolved bindings and installation data *before* swapping ECS state, so an invalid tuned preset never clears the current mode, resolved bindings, action children, current-frame `OrbitCamInput`, or emits a replacement/success event. Reflected editing mirrors the runtime shape under `reflect-input-modes`: `OrbitCamInputModeDraft::Preset(OrbitCamPresetDraft)` with per-preset draft structs and gain-payload drafts (defaulting `1.0`); draft→runtime conversion is the validation boundary (rejects negative/NaN/infinite) and reuses the same fallible replacement path. Runtime-to-draft export uses authored storage (not enabled-only runtime views), so disabled entries stay round-trippable. Descriptor ownership is write-only: `OrbitCamInputModeDescriptor` is not synced back from runtime state. Registration under `reflect-input-modes` covers the draft/runtime/kind types plus the concrete `OrbitCamBindingWithInputGain<...>` and `OrbitCamTouchBindingConfig` payload types. When a mode change disables active sources, debounced lifecycle state (interaction reports, settle deadlines, latches, panel highlights, animation-interrupt state) flushes on the next frame.

**6. Effective-row / effective-slow-mode derivation (`input/control_summary.rs`).**
Control summaries describe only *effective enabled* bindings, iterating `enabled_entries()`. `effective_control_rows(&OrbitCamBindings)` produces the visible OrbitCam rows. `effective_slow_mode(&OrbitCamBindings)` returns the slow mode only if it is configured *and* at least one effective control row exists; `effective_free_slow_mode(&FreeCamBindings)` does the same via `has_effective_free_controls` (true when any enabled translate/look/roll entry exists). `FreeCamRollControl` (`Enabled`/`Disabled`) is derived from `camera.roll.limit()` (a near-zero clamp means horizon-locked) and is a display concern only — it does not enter the effective-slow-mode decision. `describe_free_bindings` and `describe_free_translate_rows`/`describe_free_roll_rows` build FreeCam rows and the slow-mode hint from these enabled views.

**7. Latch closeout (`input/routing/latches.rs`).**
`clear_latches_on_mode_replaced` (reacting to `CameraInputModeReplaced`) always clears the `CameraInputSourceLatches` entry for the camera, and clears the `CameraSlowModeLatches` entry unless `effective_slow_mode`/`effective_free_slow_mode` reports an effective slow mode — the **same predicate** the panel uses, so latch and display cannot disagree.

**8. Fairy Dust panels (`crates/fairy_dust/src/camera_control_panel/`).**
`snapshot.rs` consumes the `CameraControlSummary` (rows + `slow_mode_binding_label`) produced by `describe_orbit_camera_controls`/`describe_free_cam_controls`. Displayed rows come from the effective summary, never recomputed from authored arrays. The active-source highlight is derived per-frame from reported interaction sources (`sources.intersects(row.camera_interaction_sources())`), with no separate latched highlight state. Builder wrappers (`SprinkleBuilder`, `PrimitiveBuilder`, `StudioLightingBuilder`, `CameraHomeBuilder`, `TitleBarBuilder`) accept `impl Into<OrbitCamPreset>` and install via `OrbitCamInputMode::with_preset`; preset cycling (`preset_switch.rs`) compares by `preset.kind()` and constructs default target presets only on an explicit cycle.

**Master gain (`orbit_cam/controller.rs`, `input/axis_response.rs`).**
The controller multiplies resolved semantic input by the axis `Sensitivity` last: `camera_input.orbit = input.orbit().pixels() * orbit_cam.orbit.sensitivity()` (and likewise pan, `scroll_line`, `scroll_pixel`). `Sensitivity` lives on each `AxisResponse` inside the `OrbitCam::{orbit,pan,zoom}` operations and is the final master gain, downstream of all input gain.

## Invariants

- **Per-device tuning lives in binding/preset lowering, not camera math.** Input gain is stamped onto binding/config payloads by presets and applied at the binding/adapter boundary; do not move it into `OrbitCam`/`FreeCam` or the controller.
- **`OrbitCam::{orbit,pan,zoom}` axis `Sensitivity` remain the final master gains.** They are applied last in `controller.rs`; input-gain changes must not alter them, and vice versa.
- **`InputGain` is finite and non-negative; `0.0` disables runtime participation but the payload stays round-trippable.** Disabled entries are filtered by `is_enabled()`/`enabled_*` accessors out of install, source attribution, control rows, pan-overrides-orbit, pinch suppression, latches, and animation interrupts — but remain in authored storage and reflected drafts for export.
- **`InputBindingScale` stays signed.** It encodes direction and axis/unit mapping; never force it non-negative. Input gain is a separate, non-negative factor multiplied through `install_modifiers()`.
- **Native gain has one composition owner: `InputBindingEntry::install_modifiers()`.** Preset lowering must not pre-multiply `InputBindingScale`, or the gain double-applies at install.
- **Smooth-scroll gain has exactly one owner.** Selection carries the chosen binding's `input_gain`; the installed `Scale` applies it once (orbit/pan `input_gain`; zoom `PIXEL_SCROLL_SCALE * input_gain`). Injection stays raw; resolution does not re-scale. Gain must not change which target wins.
- **Preset identity is setting-insensitive.** Tuned presets label as `Preset / <kind>` via `kind().name()`; `to_bindings()` lowers the full payload for rows. `PartialEq` on `OrbitCamPreset` is full value equality; `kind()` is used only for labels, comparisons, and cycling. Custom bindings stay `Bindings / Custom`.
- **Replacement validates before replacing.** Both runtime reconciliation and reflected apply lower+validate to resolved bindings/installation first; an invalid tuned preset leaves the previous mode, resolved bindings, installation entities, action children, current-frame input, and events intact and emits no success/replacement event.
- **One effective-slow-mode predicate drives both latch cleanup and panel display** (`effective_slow_mode` / `effective_free_slow_mode`). They cannot diverge.
- **"Smooth scroll" is the public term** for Bevy pixel-scroll input; user-facing prose says "smooth scroll," not "trackpad."

## Calibration / gotchas

- **Built-in Blender-like is *not* an all-disabled slow-mode case.** Presets leave pinch (and touch) input gain at `1.0`, and `BlenderLike` still emits an enabled `OrbitCamPinchZoom`. Zeroing its mouse and smooth-scroll gain leaves pinch enabled, so the slow-mode hint correctly stays visible. To test the "slow mode configured but no effective control" path, use a **custom `OrbitCamBindings`** with `slow_mode()` set and all controls disabled.
- **"Effective slow mode" = slow mode configured AND at least one enabled contribution it can scale** — an enabled effective control row (OrbitCam) or an enabled translate/look/roll entry (FreeCam). Slow mode with everything disabled hides the row and clears active slow-mode display state.
- **`FreeCamRollControl` (horizon-lock) is derived from `camera.roll.limit()`, not from bindings, and does not enter the effective-slow-mode decision.** The shared slow-mode helper is bindings-only, so latch and display stay consistent by construction.
- **Slow-latch closeout is tested by directly triggering `CameraInputModeReplaced`**, matching the codebase convention (OrbitCam and FreeCam twins in `routing/mod.rs`), not by driving a full preset-apply pipeline. The frame-flush of debounced state is proven separately (`zero_sensitive_mode_replacement_flushes_debounced_state` in `lifecycle.rs`).
- **Panel highlight needs no dedicated closeout test.** It is derived each frame from reported interaction sources, which the lifecycle flush test already proves empty on the next frame.
- **Builder-wrapper coverage is compile-only.** Constructing `TitleBar`/`CameraHome` builders through a real `App` enters the screen-panel/rendering plugin path and needs shader assets; use compile-time type/function-pointer assertions to prove `impl Into<OrbitCamPreset>` acceptance instead.
- **Repeated singleton adapter builder calls are last-write-wins** (documented and tested), for wheel/pinch/button-drag/touch descriptors.
- **Gamepad base scale knobs (`orbit_scale`, `slow_*_scale`, `stick_dead_zone`, …) are preset payload settings.** Tuning them needs a preset-returning path (`OrbitCamGamepadPreset` scale/dead-zone setters, `into_preset()`); otherwise a tuned gamepad caller falls back to `Bindings / Custom`. `gamepad_input_gain` multiplies every generated gamepad entry (normal and gated slow) while preserving gates, `ControlSpeed`, and fast/slow ratios; `0.0` disables all matching entries.
- **Fairy Dust preset-cycle tests need `app.finish()`** after `LagrangePlugin` setup so enhanced-input context resources exist before camera-context observers run.

## Why

- **Input gain lives in the binding layer** because device calibration is a property of *how a device's raw signal maps to intent*, which is exactly what bindings/presets already model. Keeping it there means camera operation code stays device-agnostic and every source (wheel, pinch, touch, stick, drag) is tuned uniformly through one authored value.
- **The `InputGain` vs `Sensitivity` split is the mechanism, not just naming.** Reserving `Sensitivity` for the camera-axis master gain and introducing `InputGain` for the binding multiplier makes the two-stage pipeline explicit and prevents future edits from conflating "how hard this device pushes" with "how strongly this axis responds."
- **`OrbitCam` master gains stay the final multiplier** so apps keep one obvious top-level knob per axis, and per-device tuning composes *under* it rather than fighting it.
- **Preset identity is setting-insensitive** so tuning a preset does not turn it into an unrecognizable custom binding set: panels, cycling, and reflected export all key off `kind()`, and a tuned Blender-like preset still reads and round-trips as `Preset / BlenderLike`.
- **`0.0` disables but preserves** so an editor or reflected draft can turn a source off and back on without losing its authored value — runtime paths simply skip disabled entries via enabled-only views while authored storage stays complete.
- **Smooth scroll has a single scaling owner** because it is a hybrid adapter/BEI path where the same pixel-scroll signal passes through injection, selection, and installed BEI `Scale`; applying gain in more than one of those would silently square the multiplier.
- **Replacement validates before replacing** so a bad tuned preset (from code or an editor) can never destroy a working installation mid-frame; the validated lowering is computed first and swapped in only on success.
